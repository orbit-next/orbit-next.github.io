orbit-next

orbit-next is a collection of projects aimed at developing the next generation of Orbit space simulation software.

Other projects have attempted to create a replacement for the current OrbitV suite of programs. Development of these projects began from scratch, making it difficult to fully capture the complex and often undocumented behavior of OrbitV. The incompatibility caused by this approach poses a serious challenge to adoption.

In orbit-next, we work directly with the existing OrbitV source code to create orbitc, a 100% compatible, drop-in replacement for OrbitV with identical behavior.

orbitc will be a starting point for further development, eventually aimed at creating a full replacement for OrbitV. In addition to inherent backwards compatibility, this approach also provides developers the opportunity of fully understanding the internals of the currently used OrbitV, a requirement for implementing a useful successor.

Current projects

orbitc

git repository

orbitc is a faithful, line-by-line port of OrbitV programs to C, as well as a custom written runtime library to support the ported programs.

C was chosen as the target language due to its wide availability, support of low-level constructs such as setjmp and goto, necessary for a reasonably straightforward translation, and a wide range of debugging and analysis tools.

orbitc by itself does not provide the facilities necessary for displaying graphics. Instead, it outputs a stream of graphics commands to be interpreted by a graphics adapter, which can be independently implemented.

orbit-ws

git repository

orbit-ws is a graphics adapter for orbitc consisting of a Node.js server and a browser-based client, communicating via websockets.

Graphics commands are streamed from an orbitc program to the browser-based client, which executes them and displays the result using an HTML5 <canvas>. Keyboard input is streamed from the client to the orbitc program.

The orbit-ws architecture inherently allows for the client and server to run on different machines in a network. In the future, capabilities can be added that allow multiple users to interact with a single orbitc program, potentially with some users entering input and others only viewing output.
